# Architecture Revision Audit

This document records the weak areas in the initial blueprint revision and the concrete corrections now encoded in the repository.

| Weak area | Why it was weak | Concrete fix | Applied artifact |
| --- | --- | --- | --- |
| Repository boundaries were too broad | `apps` and `packages` were named, but domain and infrastructure responsibilities were not explicit enough for a multi-team production codebase | Split the repository blueprint into runtime apps, shared packages, docs, and ADRs with named responsibilities and dependency rules | `docs/architecture/repository-blueprint.md` |
| Shared contract ownership was vague | API, web, worker, and realtime surfaces had no concrete shared schema boundary | Added `@megaconvert/contracts` with typed schemas for IDs, platform responses, messaging, meetings, errors, and realtime envelopes | `packages/contracts/**` |
| Validation strategy was under-specified | The previous blueprint mentioned validation but did not encode it into a reusable implementation boundary | Added `@megaconvert/config` with strict environment parsing, typed outputs, and structured error formatting | `packages/config/**` |
| Runtime safety for bootstrapping was weak | The API read raw environment variables and the worker runtime did not exist | Added typed runtime loaders and a worker bootstrap with fatal error handling, signal handling, and heartbeat logging | `apps/api/src/main.ts`, `apps/worker/**` |
| Event-driven architecture lacked a concrete execution boundary | The earlier version referenced outbox and workers but did not establish a real worker application boundary | Added a dedicated `apps/worker` runtime so asynchronous work has a first-class process boundary from day one | `apps/worker/**` |
| Meeting architecture was too coarse | "Use LiveKit" alone was not enough to define system ownership and integration boundaries | Added an ADR that splits control plane, media plane, policy, and meeting-linked conversation responsibilities | `docs/adr/0002-meeting-and-media-plane-boundary.md` |
| Search design was too shortcut-prone | Starting with PostgreSQL search was acceptable, but the abstraction boundary was not explicit enough | The repository blueprint now defines search writer/reader boundaries so the first adapter can be PostgreSQL without coupling the product to it | `docs/architecture/domain-boundaries.md` |
| Security guidance was not translated into build surfaces | Security requirements existed in prose but were not tied to repository responsibilities | The blueprint now assigns security policy, audit, auth, validation, upload safety, and abuse controls to explicit modules and runtime layers | `docs/architecture/runtime-topology.md`, `docs/architecture/domain-boundaries.md` |
| Execution sequencing lacked measurable exit criteria | Phases existed, but they were not strong enough for staff-level delivery planning | Replaced loose milestones with phase objectives, acceptance gates, and blocked/unblocked sequencing | `docs/architecture/execution-plan.md` |
| Naming quality needed to be stricter | Some names were generic and did not communicate ownership clearly enough | Tightened package and document names around contracts, runtime topology, repository blueprint, and domain boundaries | `docs/architecture/**`, `packages/**` |
| Integration completeness was insufficient | The previous blueprint did not wire the improved boundaries into the current apps | The web app and API now consume typed environment loaders, and the API returns typed platform shapes | `apps/web/src/lib/public-environment.ts`, `apps/web/src/app/page.tsx`, `apps/api/src/app.service.ts`, `apps/api/src/main.ts` |
| Maintainability and extensibility rules were implicit | Teams need explicit dependency and layering rules to prevent architecture drift | The revised blueprint defines allowed dependency directions, bounded contexts, and extraction rules for future service decomposition | `docs/architecture/repository-blueprint.md`, `docs/adr/0001-runtime-architecture.md` |
